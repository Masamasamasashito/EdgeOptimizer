## 前提：EdgeOptimizer が持つ既存の強み（Web版から継承可能）

| Web版の能力 | LLM版への転用可能性 |
|---|---|
| GEO分散リクエスト（AWS/Azure/GCP/CF × 複数リージョン） | 最寄りの推論エンドポイントへのジオルーティング |

## 各エンジンが「手が届かない」課題

### 🦙 llama.cpp
| # | 痛みポイント | 詳細 |
|---|---|---|
| 1 | **高並列の壁** | Continuous Batching が弱く、同時数ユーザー増加で TTFT が指数爆発。`n_parallel` の最適値を外部から自動探索する手段がない。 |
| 2 | **エンドポイント健全性の欠如** | 組み込み `/health` は存在するが、KVキャッシュ使用率・実効 TPS・GPU温度などの **LLM専用メトリクス** を一箇所で可視化する標準的な外部ツールがない。 |
| 3 | **テナント間分離** | シングルプロセス前提なので、複数プロジェクト・複数ユーザーに対するリクエスト分離・認証が 自前実装になる。 |
| 4 | **GEO分散との組み合わせ** | エッジ（ラップトップ / Jetson / Mac Studio 等）に llama.cpp を複数配置したとき、どのノードに投げるか決める **インテリジェントルーター** が存在しない。 |

---

### 🚀 vLLM
| # | 痛みポイント | 詳細 |
|---|---|---|
| 5 | **クロスリージョン KV キャッシュ** | 同一セッションのユーザーが別リージョンにルーティングされると KV キャッシュが失われ、プリフィルを 0 から再計算。KV キャッシュを地理的に同期・転送する仕組みがない。 |
| 6 | **プレフィックス認識ルーティング不在** | 同一システムプロンプトを持つ複数リクエストを「KV キャッシュが温まっているノード」に優先送信する機能は vLLM 単体にはない（別途ゲートウェイが必要）。 |
| 7 | **コールドスタート計測の不透明性** | スケールアウト時のコールドスタートレイテンシ（モデルロード時間）を事前に可視化・予測する ツールがない。スケールゼロからの復帰遅延が予測不能。 |
| 8 | **マルチエンジン A/B テスト困難** | vLLM v0.5 と v0.7 の差異、または vLLM と SGLang を同一トラフィックで比較する仕組みが公式にない。 |

| # | 痛みポイント | 詳細 |
|---|---|---|
| 9 | **RadixAttention の恩恵測定が困難** | キャッシュヒット率がどのくらい改善しているか、ワークロードごとに可視化するツールが成熟していない。 |
| 10 | **ユニークプロンプトへの劣化検知** | プレフィックス共有度が低いトラフィックが混入したとき vLLM より遅くなる可能性があるが、それをリアルタイムで検知して **自動的に vLLM へフォールバック** する仕組みがない。 |

| # | 痛みポイント | 詳細 |
|---|---|---|
| 11 | **コンパイル後の性能予測差** | TRT エンジンをビルドした後、実際のリアルトラフィックで期待通りのスループットが出るか **事前に外部から検証する手段** がない。デプロイ後に初めて問題を発見する。 |

## EdgeOptimizer が「痒いところに届く」理由と構想

```
既存 EdgeOptimizer (Web版) の本質
  ↓
  「GEO分散した測定ノードから、あるエンドポイントの性能を多角的・定量的に計測する」

LLM版への置き換え
  ターゲット: CDN Edge → 推論エンドポイント (llama.cpp/vLLM/SGLang/TRT-LLM)
  バリアント: UA/Lang → プロンプト/量子化/エンジン/コンテキスト長
  メトリクス: Cache-Control/TTFB → TTFT/ITL/KVキャッシュヒット率/Token/s
  セキュリティ: 4層セキュリティ → 推論ゲートウェイ認証統一
```

### EdgeOptimizer LLM版が提供できるユニークな価値

| 価値 | 既存エンジンとの差別化 |
|---|---|
| **GEO分散プローブによるエンドポイント到達性計測** | AWS Lambda/Azure Functions/GCP CloudRun/CF Workers から推論ノードへの RTT を常時計測。ルーティング判断の基準線を提供。 |
| **エンジン横断ベンチマーク自動化** | 同一プロンプトセットを llama.cpp / vLLM / SGLang / TRT-LLM に同時投入し TTFT・TPS・コストを比較。 |
| **プレフィックス認識疑似ルーティング評価** | KVキャッシュが温まっているノードへの優先ルーティングが効いているか外部から定量検証。 |
| **コンプライアンス対応ルーティング検証** | 「EU リクエストは EU ノードに届いているか？」を実際の GEO 分散ノードから確認。 |
| **コールドスタートウォームアップ計画** | CDN のキャッシュウォームアップを LLM のモデルウォームアップに転用。スケールゼロからの復帰前に事前プローブして温める。 |
| **コスト/トークン ルーター** | Self-Hosted vs Cloud API のリアルタイムコスト比較 + 最適ルーティング切替。 |

## 🔀 ハイブリッドルーティング：軽い／重い推論の切り分けロジック

### アーキテクチャ概要

```
ユーザーリクエスト
  │
  ▼
┌──────────────────────────────────┐
│  Cloudflare Workers              │  ← ルーティング/認証/キャッシュ/レート制限
│  (EdgeOptimizer Gateway)         │     下記判定ロジックを実行
└──────────┬───────────────────────┘
           │ 判定結果により分岐
     ┌─────┴──────┐
     │            │
     ▼            ▼
[軽い推論]    [重い推論]
CF Workers AI  GKE Inference Gateway
  TTFT<50ms    + vLLM / SGLang / TRT-LLM
               （マルチリージョン）
```

### 切り分け判断ロジック（優先順位順）

```
リクエスト受信
  │
  ├─ ① モデル指定が CF カタログ外？ → GKE へ（カスタム/FTモデル）
  │
  ├─ ② プロンプトのトークン推定 > 8,000？ → GKE へ（長文コンテキスト）
  │
  ├─ ③ max_tokens > 2,048？ → GKE へ（長文生成）
  │
  ├─ ④ X-Data-Residency ヘッダあり？ → GKE 特定リージョン強制（データ主権）
  │
  └─ ⑤ Workers AI エラー率/レイテンシ閾値超過？ → GKE へ自動フォールバック
```

### Cloudflare Workers 実装イメージ（TypeScript）

```typescript
export default {
  async fetch(request: Request, env: Env): Promise<Response> {
    const body = await request.json() as InferenceRequest;

    if (shouldRouteToGKE(body, request)) {
      // 重い推論 → GKE Inference Gateway へ転送
      return forwardToGKE(request, body, env);
    }

    // 軽い推論 → Workers AI でローカル処理（TTFT < 50ms）
    return runWorkersAI(body, env);
  }
};

function shouldRouteToGKE(body: InferenceRequest, req: Request): boolean {
  // ① カスタムモデル（CF カタログ外）
  const cfCatalog = ["@cf/meta/llama-3.3-70b-instruct-fp8-fast", /* ... */];
  if (!cfCatalog.includes(body.model)) return true;

  // ② 長文コンテキスト（トークン推定）
  const tokenEstimate = estimateTokens(body.messages);
  if (tokenEstimate > 8000) return true;

  // ③ 長文生成
  if ((body.max_tokens ?? 0) > 2048) return true;

  // ④ データ主権
  const residency = req.headers.get("X-Data-Residency");
  if (residency && residency !== "any") return true;

  return false; // → Workers AI で処理
}
```

### ユースケース別 切り分け早見表

| ユースケース | トークン目安 | 判定 | 処理先 |
|---|---|---|---|
| チャットbot 短文Q&A | 〜500 | 軽い | CF Workers AI |
| RAG 検索結果まとめ | 〜3,000 | 軽い | CF Workers AI |
| 長文ドキュメント要約 | 20,000〜 | 重い | GKE |
| コードベース全体解析 | 50,000〜 | 重い | GKE |
| カスタムFTモデル | 問わず | 重い（モデル不一致） | GKE |
| EU顧客データ処理 | 問わず | 重い（データ主権） | GKE EU リージョン強制 |

### EdgeOptimizer の真の価値

> この「切り分けルールの最適閾値」を定量的に発見するのが EdgeOptimizer の核心的役割。
> 「トークン閾値を 4,000 にした場合と 8,000 にした場合で、全体の TTFT・コスト・エラー率はどう変わるか？」
> → GEO分散プローブが実トラフィックを流して計測し、最適パラメータを算出する。

## 🏗️ EdgeOptimizer にとっての最適アーキテクチャ

```
ユーザーリクエスト
  │
  ▼
┌──────────────────────────────┐
│  Cloudflare Workers          │  ← ルーティング/認証/キャッシュ/レート制限
│  (EdgeOptimizer Gateway)     │     軽量推論はここで完結（Workers AI）
└──────────┬───────────────────┘
           │ 重い推論のみ転送
           ▼
┌──────────────────────────────┐
│  GKE Inference Gateway       │  ← KVキャッシュ認識ルーティング
│  + vLLM / SGLang / TRT-LLM   │     カスタムモデル / 大規模推論
│  (マルチリージョン)             │
└──────────────────────────────┘
```

### なぜこの構成が「アツい」のか

1. **Cloudflare Workers = EdgeOptimizer の既存資産がフル活用可能**
   - 既にCFワーカーのデプロイパイプラインを持っている
   - Workers AI の OpenAI 互換 API で推論プローブがそのまま書ける
   - 180都市からの GEO 分散計測が即実現

2. **GKE Inference Gateway = 痛みポイント #6, #13 の解**

# 【補足】専門用語解説

### 🚀 LLM 推論性能に関する指標
* **TTFT (Time To First Token):** リクエストが送信されてから、最初の1トークンが出力されるまでの時間。ユーザーが「応答が始まった」と感じる体感速度に直結する。
* **TPS (Tokens Per Second):** 1秒間に生成されるトークン数。推論エンジンのスループット（処理能力）を示す。
* **ITL (Inter-Token Latency):** トークン間隔。最初のトークンが出た後、次のトークンが出るまでの待ち時間。これが一定であるほど、スムーズなストリーミング表示が可能。
* **KV Cache (Key-Value Cache):** 過去のトークンの計算結果を保存しておくメモリ領域。これを利用することで、同じプロンプトの再計算を回避し、高速化を実現する。
* **Prefill (プリフィル):** 入力プロンプトを一度に処理し、KVキャッシュを構築するフェーズ。計算負荷が高い。
* **Decoding (デコーディング):** 生成されたトークンに基づいて、次のトークンを1つずつ順番に生成していくフェーズ。

### 🌐 ネットワーク & インフラに関する指標
* **RTT (Round Trip Time):** データパケットが送信元から宛先に到達し、応答が戻ってくるまでの往復時間。ネットワークの遅延（レイテンシー）の基本指標。
* **GEO 分散 (Geo-Distribution):** 地理的に離れた複数のリージョン（東京、シンガポール、米国など）にサーバーを配置すること。ユーザーに最も近い場所から応答を返すことで、ネットワーク遅延を最小化する。
* **Anycast:** 単一のIPアドレスに対して、ネットワークの構成に応じて物理的に最も近いサーバーへ自動的にルーティングする技術。
* **Cold Start (コールドスタート):** 停止していたサーバーレス関数や、ロードされていないモデルを起動する際に発生する、初期の起動遅向。

### 🧠 推論エンジン特有の技術
* **Continuous Batching:** 複数のリクエストを、生成が終わったものから順次入れ替えて効率的に処理する技術。vLLM などのモダンなエンジンの中核。
* **RadixAttention:** SGLang 等で採用されている、プロンプトの共通部分（Prefix）をツリー構造で管理し、KVキャッシュの再利用率を極限まで高める技術。
* **Quantization (量子化):** モデルの重みの精度（例: 16bit → 4bit）を下げることで、メモリ使用量を削減し、推論速度を向上させる技術。
* **Speculative Decoding (投機的デコーディング):** 小さな軽量モデルで先にトークンを予測し、大きなモデルでその妥当性を検証することで、高速に生成を行う手法。
