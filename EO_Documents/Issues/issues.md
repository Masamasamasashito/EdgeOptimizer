# Edge Optimizer アーキテクチャ 厳しい評価レポート

> 作成日: 2026-03-10

---

## まず本当に優れている点（前提確認）

- **Request Engine Core の設計は本物**。`_build_flat_result` の出力スキーマ設計、拡張機能レジストリ、CDN検出ロジックは業務レベルを超えている
- **マルチクラウド非依存化**の思想は正しい
- **DNS TXT認証**による不正リクエスト防止は見落とされがちで、評価できる

---

## 🔴 厳しい批判：アーキテクチャ上の構造的問題

---

### 問題1: n8n がローカル前提 = 単一障害点 (SPOF)

```
n8n（ローカルDocker） → AWS Lambda / Azure / GCP / CF
```

**批判:** n8nがダウン・PCが落ちた瞬間にEO全体が止まる。

- スケジューラ（Cron）もn8n任せ
- HA構成（高可用性）の概念がゼロ
- 運用しているのが1人なら今は許容できるが、**製品・サービスとしては致命的**

> *「設計に時間をかけたコアロジック」と「運用できるシステム」は別物*

---

### 問題2: セキュリティモデルが静的すぎる

```python
return hashlib.sha256(f"{url}{secret}".encode()).hexdigest()
```

**批判:**

- `secret` が漏れたら全リクエストが偽装可能で、しかも**シークレットのローテーション機構がない**
- HMAC ではなく SHA-256(url + secret)の単純結合 → **長さ拡張攻撃**への理論的な脆弱性
- タイムスタンプ付き署名（有効期限）がない → **リプレイ攻撃**が可能
- n8nの `N8N_EO_REQUEST_SECRET` は `.env` で管理 → Gitに混入リスクがゼロではない

---

### 問題3: 観測可能性（Observability）の完全な欠如

**批判:**

- ワークフロー全体の**実行ログが構造化されていない**（n8nのGUI内ログだけ）
- Lambda/Azure/GCP/CF各エンジンの**エラー率・成功率をリアルタイムで把握する手段がない**
- CloudWatchやApplication Insightsの活用がドキュメントに一切言及なし
- 「CSVが出力されていれば正常」という確認方法では、**サイレント障害**（一部リクエストが静かに失敗）を検知できない

---

### 問題4: データパイプラインが脆弱すぎる（EA含む）

```
Lambda → Google Drive → n8n → Google Sheets → Looker Studio
```

**批判:**

- Google Sheets の**セル上限（1000万セル）**を長期運用で必ず踏む
- Google Drive経由のファイル増殖で**ファイル管理が破綻**する（命名規則だけでは限界）
- n8nが処理しそこなったCSVは**永遠に再処理されない**（冪等性・再送機構なし）
- 途中でn8nが落ちたら**データの欠損が無言で発生**する

---

### 問題5: テスト戦略が存在しない

**批判:**

- `request_engine_core.py` に**ユニットテストが一切ない**
- CDN検出ロジック (`_detect_cdn`) は複雑かつ条件分岐が多い → テストなしでの保守は危険
- GitHub Actionsのデプロイ前にlint/testが走っていない
- 「動いていれば正しい」という運用になっている

---

### 問題6: n8nワークフローがコード管理されていない

**批判:**

- `eo-n8n-workflow-jp.json` はGit管理されているが、**実際にn8nに入っているワークフローとの乖離を検証する手段がない**
- ノード設定（URLや認証情報）がn8nのGUI内に閉じている → **IaCとして不完全**
- ワークフローに変更を加えた場合、JSONをエクスポートして手動でGitにコミットしなければならない → 人的ミスが発生しやすい

---

### 問題7: スケーラビリティの天井が低い

現状の構造では：

- URLが数千→数万になったとき n8n の**メモリ使用量が爆発**する可能性
- `#340 Random Sleep (1000-3999ms)` をシリアルに挟むため、**URL数に比例して実行時間が線形増加**
- 並列化の設計が不明確（n8nのbatch処理はメモリ効率が悪い）

---

## 🟡 軽度の問題（改善余地あり）

| 問題 | 説明 |
|------|------|
| CDN検出の優先順位 | Primary検出で最初マッチしたもので`break`するが、複数CDNが重なるケース（例: Cloudflare + Vercel）で誤検知しうる |
| `_get_tls_version` の信頼性 | `response.raw._connection.sock` へのアクセスはhttpx等への移行で壊れる |
| HTTP/3未対応 | `requests` ライブラリはHTTP/3非対応。将来的に測定値が不正確になる可能性 |
| `from_area` の検証なし | 不正な文字列が入っても弾かれない |

## 🔵 本質的な結論

> **EOは「個人が自分のサイトを管理するための洗練されたツール」としては非常によくできている。しかし「他者に提供するプロダクト」「長期安定運用するシステム」として見ると、運用・信頼性・セキュリティの層が根本的に薄い。**

設計にかけた時間はコアロジックに集中しており、**インフラ・運用・テスト・障害対応**の設計は後回しにされている。これは個人開発では仕方のない優先順位だが、**それを認識した上で次の投資先を決める必要がある**。

## 優先度別 改善ロードマップ案

| 優先度 | 課題 | 対応案 |
|--------|------|--------|
| 🔴 高 | SPOF (n8n) | n8nをクラウド版に移行 or Docker Swarm/k8s化 |
| 🔴 高 | リプレイ攻撃対策 | タイムスタンプ付きHMAC署名に変更 |
| 🔴 高 | サイレント障害 | Dead Letter Queue + アラート設計 |
| 🟡 中 | テスト不在 | `pytest` でCDN検出・flat result・トークン計算を最低限カバー |
| 🟡 中 | データ信頼性 | n8n処理済みチェック機構（処理済みファイルのトラッキング） |
| 🟢 低 | HTTP/3対応 | `httpx` への移行検討 |
| 🟢 低 | Google Sheets上限 | 月次でアーカイブシートへ移動するn8nワークフロー追加 |
