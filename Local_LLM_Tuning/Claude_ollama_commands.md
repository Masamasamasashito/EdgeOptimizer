# ローカルモデル指定claudeコマンドの基本

- 【入門記事】ClaudeCodeの中級者になりたい人は集合
    - https://qiita.com/K5K/items/72cc4282819ace823524

- LLM（Large Language Model / 大規模言語モデル）本質は「入力された文章に対し、確率的に最も自然な続きの文字列を返す」こと
- LLM単体は文章生成しかできない。ファイルを読んだり、コマンドを実行したり、外部サービスを操作したりする能力はない。
- Function Calling（Tool Use）で、LLMに「使えるツール一覧」を渡しておくと、LLMが状況に応じて適切なツールを選び、呼び出すことができる。
- LLM + Function Callingの組み合わせで、文章生成だけでなく「行動」が可能。
- コンテキスト・・・AIに渡す情報,文脈のこと
    - 会話の流れ、指定したファイルの中身、使えるツールの一覧などURLの内容、MCPやツールの実行結果など。
- コンテキストウィンドウ・・・AIが一度に扱える情報量の上限
    - 超えるとエラーや古い情報参照できなくなる
    - ClaudeCodeはコンテキストウィンドウ上限に近づくと `/compact` コマンドで自動的にコンテキスト圧縮する

# LLM実行時にコンテキストが扱われるデバイス

1. 入力「コンテキスト」は RAM に読み込まれる
2. 計算開始直前、データが VRAM へ送られる
3. GPUが VRAM 上データで計算、回答を出す

「VRAMが足りない」と言われるのは、「超高速な作業台」にデータが乗り切ってない状態。

|項目|RAM (Main Memory)|VRAM (Video RAM)|
|:---:|:---:|:---:|
|主な利用者|CPU (PCの脳)|GPU (画像・計算の専門家)|
|役割|OSやアプリ全体の作業場|映像描写・AI計算の専用作業場|
得意なこと|複雑な事務処理・複数アプリの管理|膨大な単純計算（行列演算など）|
|データ転送速度|高速|圧倒的に高速 (帯域が非常に広い)|
|物理的な場所|マザーボードのスロットに刺さる|グラフィックボードの中に内蔵|
|役割イメージ|広い事務机|超高速な専用作業台|

# MCP（Model Context Protocol）

- ClaudeCodeと外部ツール・サービスを連携させる仕組み
- Slack、GitHub、データベース等をMCPサーバーとして設定、ClaudeCodeが直接操作できるようになる
- MCPを設定すると、ツール実行結果がコンテキストに追加され、コンテキストを消費する

# Skills

- ClaudeCodeにタスクの手順やルールを教えるための仕組み
- マークダウンファイルで「業務マニュアル」を作成し、`.claude/skills/` フォルダに置く
- 必要なときに自動で呼び出される
- Skillsは呼び出されたときのみコンテキストを使う(Progressive Disclosure)
- コンテキスト効率の高さから、ClaudeCodeはSkillsが中心的な役割を担っている

# RAM と VRAM の消費理解

## LLM 推論時のメモリ消費フロー

┌─────────────┐    ┌─────────────┐
│   RAM       │───▶│   VRAM      │
│ (メインメモリ) │    │ (ビデオメモリ) │
└─────────────┘    └─────────────┘
    │                     │
    ├── OS/シェル環境     ├── モデル重み（固定）
    ├── ファイルシステム  ├── KV キャッシュ（可変）
    └── コマンドバッファ   └── 作業領域（一時）

## RAM の消費（主に起動前・システム側）

- **Ollama サーバープロセス**: ~50-200MB
- **モデルファイルのディスク→RAM 転送**: 推論開始時のみ
- **コンテキスト構築**: ファイル読み込み、会話履歴準備

**RAM は通常ボトルネックにならない**（現代PCなら8GB以上あれば余裕）

## VRAM の消費（推論時の本番領域）

### アーキテクチャ層：CPU/GPU メモリ階層の物理構造

#### CPU のキャッシュ階層（RAM アクセス経路）
```
CPU コア → L1 キャッシュ (32-64KB, 1 クロック)
           ↓ ミス時
         L2 キャッシュ (256KB-1MB, 数クロック)
           ↓ ミス時
         L3 キャッシュ (10-30MB, 10-20 クロック)
           ↓ ミス時
         RAM (DDR4/5: 25-84GB/sec, 100ns レイテンシ)
```

**特徴**:
- **逐次アクセス特化**: CPU は複雑な分岐処理を得意とするが、メモリアクセスは逐次実行
- **キャッシュミス即死**: L3 ミスで RAM にアクセスすると数百サイクルロス

#### GPU のメモリ階層（VRAM アクセス経路）
```
SM (Streaming Multiprocessor) → レジスタファイル (数 KB, 専用化)
                                ↓
                             共有メモリ (64-128KB, 超高速)
                                ↓
                          L2 キャッシュ (2-4MB)
                                ↓
                         VRAM (GDDR6X: 500-1TB/sec, HBM3: 1.6TB/sec)
```

**特徴**:
- **大規模パラレル特化**: 数千コアが同時にメモリアクセス
- **帯域幅が命**: VRAM バンド幅 = 推論速度を直接決定

#### PCIe バス：CPU と GPU の橋渡し
```
[CPU] ←→ [PCIe バス] ←→ [GPU]
          (x16 Gen3: 16GB/sec)
          (x16 Gen4: 32GB/sec)
          (x16 Gen5: 64GB/sec)
```

**ボトルネックポイント**: モデルを RAM → VRAM に転送する際、このバス帯域が限界

| メモリ種 | バス帯域 | レイテンシ | 用途 |
|:---|:---:|:---:|:---|
| CPU L1/L2 | 数 TB/sec | <1ns | 頻繁な変数・ループカウンタ |
| VRAM (GDDR6X) | 500-1000GB/sec | ~100ns | モデル重み・大規模行列 |
| PCIe x16 Gen4 | 32GB/sec | ~10μs | CPU↔GPU データ転送 |

---

### 1. モデル重み（固定、起動時読込）
VRAM = パラメータ数 × 量子化サイズ + 余白

Qwen3.5:27B (INT4) = 27GB × 0.5 × 1.1 ≈ 15GB
DeepSeek-R1:32B (INT4) = 32GB × 0.5 × 1.1 ≈ 18GB
Gemma3:12B (INT4) = 12GB × 0.5 × 1.1 ≈ 7GB

### 2. KV キャッシュ（可変、トークン増加で肥大化）
- **コンテキストごとに蓄積される中間結果**
- VRAM 枯渇の主要因 → 「長文処理でアウト」する正体
- 削減不可能（モデル設計上必須）

KV キャッシュ ≈ コンテキスト長 × レイヤー数 × 隠れ層の次元数 × バッチサイズ

実践的目安:
├─ 1K トークン (短会話)   → +0.5GB
├─ 8K トークン (中ファイル) → +2-4GB
├─ 32K トークン (大規模文書) → +8-12GB
└─ 128K トークン (超大文書)  → VRAM 枯渇確実

---

### 推論プロセス層：トークン生成の物理的経路

1 つのトークンが生成されるまで、データは以下の物理経路を辿る。

#### ステップごとのメモリ移動

```
┌─────────────────────────────────────────────────────────┐
│ 入力トークン（文字列）                                  │
└─────────────────────────────────────────────────────────┘
           ↓
    [RAM に文字列バッファ] ← OS メモリ管理
           ↓
    [PCIe を経て GPU へ転送] ← バス帯域ボトルネック
           ↓
    [VRAM にテンソル変換] ← embedding ルックアップ
           ↓
    [VRAM 内：Transformer 層を通過] ← 行렬演算の連続
           ├→ QKV プロジェクション（VRAM 読み出し）
           ├→ Self-Attention 計算（共有メモリ利用）
           └→ Feed Forward（レジスタファイル利用）
           ↓
    [VRAM に確率分布出力] ← ソフトマックス
           ↓
    [サンプリングで次トークン決定]
           ↓
    [RAM に文字列として戻す] ← 表示/次の処理へ
```

####各环节のボトルネック

| ステップ | ボトルネック要因 | 改善策 |
|:---|:---|:---|
| RAM → VRAM 転送 | PCIe バス帯域 (32GB/sec 上限) | バッチ処理、モデルは常駐させる |
| VRAM 内計算 | VRAM バンド幅依存 | HBM/GDDR6X の高速 GPU 選択 |
| 長文処理時 | KV キャッシュ肥大化で VRAM 枯渇 | `/compact`、セッション分割 |

#### なぜ「逐次生成」が遅いのか？

LLM は**前のトークンの出力が次の入力になる**ため、並列化不可能：

```
トークン 1 生成 → トークン 2 生成 → トークン 3 生成 → ...
    (VRAM 計算)     (VRAM 計算)     (VRAM 計算)
      ↓                ↓               ↓
   必須待機         必須待機        必須待時
```

これが「10 秒で 50 トークン」といった低速推論を生む物理的制約。

- モデル構造に依存

### ボトルネック分析層：なぜ VRAM が足りないのか？

#### VRAM 枯渇の物理的理由

**1. メモリ分離アーキテクチャの制約**

CPU の RAM と GPU の VRAM は物理的に独立したメモリで、互いにアクセスできない。

```
┌───────────────┐         ┌───────────────┐
│    CPU        │         │    GPU        │
│               │         │               │
│ ┌───────────┐ │         │ ┌───────────┐ │
│ │  L3       │ │         │ │ レジスタ   │ │
│ ├───────────┤ │         │ ├───────────┤ │
│ │  RAM      │ │         │ │ 共有メモリ  │ │
│ │ (16-64GB) │ │         │ ├───────────┤ │
│ └───────────┘ │         │ │ L2 キャッシュ││
│               │         │ ├───────────┤ │
│ OS/アプリ実行   │         │ │  VRAM     │ │
│               │         │ │(12-24GB)   │ │
└───────────────┘         │ └───────────┘ │
       │                  │  モデル重み    │
       │  PCIe            │  KV キャッシュ  │
       ├─────────────────>│               │
       │ (転送ボトルネック)  └───────────────┘
```

**物理的制約**:
- GPU は VRAM のみを直接アクセス可能（RAM に手を伸ばせない）
- RAM → VRAM への転送は PCIe を経由し、帯域制限あり
- VRAM 枯渇 = モデルが載らない = 推論不能

#### VRAM バンド幅と計算速度の不一致

GPU コア自体は秒間数百 GB の計算が可能だが、VRAM からデータを取り込む速度が追いつかない：

| GPU | VRAM バンド幅 | FP16 計算能力 |
|:---|:---:|:---:|
| RTX 3060 (12GB) | 360 GB/sec | 13 TFLOPS |
| RTX 4090 (24GB) | 1,008 GB/sec | 83 TFLOPS |

**意味**: コアはデータ待ち状態になりやすい（Memory-Bound）

#### VRAM 不足時のシステム動作

VRAM が枯渇すると、以下の現象が発生：

```
[ケース 1] VRAM 完全枯渇 → プロセス強制終了 (CUDA OOM)

[ケース 2] CPU Offload → RAM にモデルの一部を退避
           ├→ PCIe 経由で必要な層のみ VRAM に転送
           └→ 推論速度 10-50 倍低下

[ケース 3] スワップ → ディスク領域への退避
           └→ SSD でも数十分遅延発生
```

#### RAM/VRAM の容量対決シミュレーション

**環境**: RTX 4060 Ti (16GB VRAM) + 32GB RAM

| モデル | VRAM 消費 | RAM 消費 | 状態 |
|:---|:---:|:---:|:---|
| Qwen3.5:8B | ~6GB | ~100MB (Ollama プロセス) | ✅ 余裕 70% |
| Qwen3.5:14B | ~9GB | ~100MB | ⚠️ 残量 43%（長文 NG） |
| Qwen3.5:27B | ~15GB | ~100MB | ⚠️ 起動可能だが VRAM 枯渇限界 |
| DeepSeek-R1:32B | ~18GB | ~100MB | ❌ 起動不可 (16GB 超過) |

**重要**: RAM が 64GB あっても、VRAM 16GB でモデルが載らない。物理的に別メモリ。

#### なぜ「長文で落ちる」のか？

**KV キャッシュの累積効果**:

```
モデル起動時：VRAM = モデル重み (固定) + 最小バッファ
  ↓ トークン生成開始
+100 トークン: +50MB
+1,000 トークン: +500MB
+8,000 トークン: +4GB ← 危険域
+32,000 トークン: +16GB ← VRAM 枯渇確定 (24GB GPU でも NG)
```

**物理的解決不可能な理由**:
- KV キャッシュは「過去の計算結果の記憶」→ 削除できない（文脈喪失）
- トークン数増 = VRAM 消費増 = プロセス終了

## コンテキストウィンドウ制御のコツ

VRAM 枯渇対策がコンテキスト管理の本質。

### トークン数の VRAM 影響

| 内容規模 | 概算トークン | +VRAM 消費 | 安全域判定 |
|:---|:---:|:---:|:---|
| 短文会話を10ターン | ~2K | +0.5GB | ✅ あらゆるモデルで可 |
| Python ファイル1個 (100行) | ~300-500 | +0.2GB | ✅ 安全 |
| 大規模ファイル群 (500 行×5) | ~2K-4K | +1GB | ⚠️ 小モデルでは注意 |
| PDF ドキュメント全文 | ~10K+ | +3-5GB | ⚠️ VRAM 8GB では NG |

### VRAM 枯渇の兆候と即座に対策

**兆候**: `CUDA out of memory` / 推論速度激減（1token/秒未満）/ Ollama プロセス再起動

| 対策 | 消費削減効果 | 使用タイミング |
|:---|:---:|:---|
| `/compact` | VRAM -30-50% | 長文処理後、会話延伸時 |
| ファイル部分読み込み | VRAM -70-90% | `line:X-Y` でピンポイント指定 |
| `/clear` + 再開始 | VRAM リセット完全 | セッション疲弊時（最も効果大） |
| モデル小型化へ切り替え | VRAM -40-60% | qwen3.5:27b → qwen3.5:8B |

### 読み込み順序の最適化（VRAM エネルギー節約）

**基本原則**: 「最小文脈から順次追加」

1. **問題定義だけ**: "この関数のバグを探して"（VRAM 最低消費で開始）
2. **該当ファイルの関連行のみ**: `read_file(lines=50-70)` でピンポイント読み込み
3. **理解が固まったら全体参照**: 必要に応じてファイル全体読ませる
4. **修正確認後 `/compact`**: 不要な履歴で VRAM 浪費を防ぐ

### VRAM 12GB 環境でのモデル使用戦略

| モデル | 固定消費 | コンテキスト最大値（目安） | 推奨使用ケース |
|:---|:---:|:---:|:---|
| qwen3.5:8b | ~6GB | 16K トークン | コードレビュー、短文生成 |
| qwen3.5:14b | ~9GB | 8K トークン | 中規模タスク（ファイル複数） |
| qwen3.5:27b | ~15GB | ❌ 起動不可能 | VRAM アップグレード必須 |

### 大規模ファイルの分割処理戦略

【例】800 行の TypeScript ファイルを修正

× ダメなやり方：
- ファイル全体読ませて「バグ修正して」
→ コンテキスト浪費 + 不特定箇所の混同

○ VRAM エネルギーなやり方:

1. 「ファイル構造だけ見せる（先頭 20 行）」→ +0.5GB
2. 「問題関数 problemFunc() の定義行を教えて」→ AI が行番号を特定
3. `read_file(line=50-120)` でピンポイント読込 → +0.8GB
4. バグ説明後、`/compact` → VRAM を 50% 削減
5. 次の問題へ（VRAM 枯渇リスク回避）

VRAM 使用量のリアルタイム監視

Windows: タスク管理「パフォーマンス」タブ → GPU 選択 → 「専用 GPU メモリ」
Linux/macOS: nvidia-smi -l 1

警戒ライン:
- 80% 超え: /compact を検討
- 90% 超え: /clear でセッションリセット
- 100%: プロセス再起動（VRAM エネルギー限界）

# ローカルollama利用のための環境変数設定

- Claudeは、デフォルトでAnthropicサーバーとの通信
- デフォルトを変えて明示的にローカルOllamaインスタンスにリダイレクトするため環境変数を設定する必要あり。
- APIキーはダミー

```
$env:ANTHROPIC_BASE_URL = "http://localhost:11434/"
$env:ANTHROPIC_API_KEY = "ollama"
```
# 不要なトラフィックを抑制

- データ使用量制限やプライバシー重視の環境での利用を想定した設定
- テレメトリや非必須API呼び出しを停止

```
$env:CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC = "1"
```
